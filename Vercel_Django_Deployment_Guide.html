Vercel & Django Deployment Postmortem

Project: Ark Global Resources

Date: June 30, 2026

This document outlines the specific challenges encountered while deploying a complex Django application to Vercel's serverless infrastructure, the root causes of the errors, and the exact solutions implemented. Keep this as a reference for future Python deployments.

1. The Trial and Error Breakdown

Error 404: Missing CSS & Images

The `.gitignore` Trap

What happened: Vercel successfully deployed the HTML, but all CSS and images (like the logo) were returning a 404 Not Found error.

Root Cause: Vercel does not automatically execute collectstatic reliably for Python. To fix it, we generated the staticfiles folder locally to push to GitHub. However, inside your .gitignore file, the rules staticfiles/ and Images/ were present. This caused Git to secretly ignore and delete your CSS and Logo files during every push.

The Fix: Removed staticfiles/ and Images/ from .gitignore, ran python manage.py collectstatic --noinput locally, and pushed the folder directly to GitHub.
Error 500: FieldDoesNotExist

The Hidden Code Typo

What happened: The homepage crashed with a generic 500 error after connecting to the fresh Supabase database.

Root Cause: When we forced DEBUG=True to reveal the traceback, we found a FieldDoesNotExist error in ark_catalog/views.py. The view queried Category.objects.only('image', 'description'), but the Category model only had a poster_image field, and no description. The local SQLite database hid this error due to old cached data, but the fresh Supabase database crashed instantly.

The Fix: Updated the query in views.py to request valid fields: Category.objects.only('id', 'name', 'slug', 'poster_image').
FATAL: EMAXCONNSESSION

Connection Pool Exhaustion

What happened: Vercel logs threw a FATAL error stating "max clients reached in session mode". The database completely locked up.

Root Cause: Serverless architecture (Vercel) spins up a new mini-server for every visitor. In settings.py, Django was instructed to keep connections open for 10 minutes (conn_max_age=600). Vercel rapidly opened 15 connections, kept them open, and instantly exhausted Supabase's Session connection pool.

The Fix: Set conn_max_age=0 in Django to force immediate connection closure. Instructed the use of Supabase's Transaction Pooler Port (6543) in Vercel environment variables.
[Errno 30] Read-only file system: '/var/task/media'

The Serverless Read-Only Crash

What happened: Attempting to edit or upload a product in the CEO Portal crashed the application entirely.

Root Cause: Vercel's physical hard drives are permanently read-only to protect against malicious attacks. Cloudinary was supposed to handle uploads, but because it was wrapped in an if not DEBUG: block, any instance where Vercel suspected debug mode was active caused Django to attempt saving files directly to the read-only Vercel hard drive.

The Fix: Removed the DEBUG condition entirely for Cloudinary. We forcefully set STORAGES["default"]["BACKEND"] = "cloudinary_storage.storage.MediaCloudinaryStorage" so that images ALWAYS bypass the Vercel hard drive and go straight to the cloud.

2. Future Checklist for Django on Vercel

To prevent these issues on future deployments, ensure your project meets the following criteria before deploying to Vercel:

  • No Background Workers: Vercel does not support background tasks (like django-q). Use Render if background workers are required.
  • Database URLs: Always use the Supabase Transaction Port (e.g., 6543) for serverless environments.
  • Close Connections: Ensure conn_max_age=0 in your database configuration.
  • Commit Static Files: Do not rely on Vercel to build static assets. Run collectstatic locally and ensure staticfiles is not in your .gitignore.
  • Force Cloud Storage: Never use FileSystemStorage in production on Vercel. Force your storage backend to Cloudinary or S3 unconditionally.
